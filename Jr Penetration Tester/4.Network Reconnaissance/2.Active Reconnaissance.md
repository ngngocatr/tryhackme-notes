# Active Reconnaissance

## 1. Introduction

### Khái niệm
- **Active Reconnaissance**: Thu thập thông tin bằng cách **tương tác trực tiếp** với mục tiêu (gửi packet, kết nối, quét dịch vụ).
- **Passive Reconnaissance**: Thu thập từ nguồn công khai, **không gửi traffic** đến mục tiêu.

### So sánh

| Passive | Active |
|---------|--------|
| Không tiếp xúc mục tiêu | Tương tác trực tiếp với mục tiêu |
| Không để lại dấu vết | Để lại log, IDS/WAF Alert, Honeypot Trigger |
| DNS, WHOIS, Shodan, Certificate Logs | Browser, Ping, Traceroute, Telnet, Netcat |

### Công cụ chính
- **Browser + DevTools**: Kiểm tra Header, JavaScript, Certificate.
- **Ping**: Kiểm tra host hoạt động, suy đoán OS qua TTL.
- **Traceroute / mtr**: Xác định đường đi của gói tin.
- **Telnet**: Banner grabbing trên dịch vụ cũ (legacy).
- **Netcat (nc)**: Banner grabbing, port probing, giao tiếp client/server.

### Thực tế hiện nay
- Hầu hết hệ thống triển khai:
  - CDN (Cloudflare, Akamai)
  - WAF
  - Zero Trust
  - SIEM / EDR
- HTTPS gần như thay thế Telnet trong web.
- IPv6 ngày càng phổ biến, ICMPv4 có thể bị chặn.

### Red Team vs Blue Team

**Red Team**
- Hòa vào lưu lượng bình thường.
- Dùng User-Agent hợp lệ.
- Gửi request chậm để giảm khả năng bị phát hiện.

**Blue Team**
- Giám sát:
  - Access Logs
  - Firewall Logs
  - WAF Events
  - IDS Alerts

### Lưu ý
- Chỉ thực hiện Active Reconnaissance khi có **ủy quyền hợp pháp** (Pentest Contract, Bug Bounty Scope).
- Quét trái phép có thể vi phạm pháp luật.

## 2. Web Browser Reconnaissance

### Vì sao dùng Web Browser?
- Có sẵn trên hầu hết hệ thống.
- Traffic giống người dùng bình thường → khó bị phát hiện.
- Phù hợp cho **Active Reconnaissance**.

### Transport-Level Basics
- **HTTP**: TCP **80** (hiện ít dùng).
- **HTTPS**: TCP **443** (phổ biến).
- **HTTP/3 (QUIC)**:
  - Chạy trên **UDP 443**.
  - Kết hợp TCP + TLS → kết nối nhanh hơn.
  - Trong DevTools > Network, protocol hiển thị **h3**.
- Có thể truy cập cổng khác:
  - `https://target.com:8443`
  - `http://192.168.1.100:8080`

### Developer Tools (`Ctrl + Shift + I`)

#### Network
- Xem:
  - Request/Response Headers
  - `Server`
  - `X-Powered-By`
  - `Content-Security-Policy`
  - Cookies
  - Status Code
  - Timing

#### Console
- Thực thi JavaScript.
- Kiểm tra lỗi và tương tác với DOM.

#### Sources
- Xem JavaScript, CSS, HTML.
- Có thể phát hiện:
  - API Endpoint
  - Directory Structure
  - Internal Service
  - Developer Comment

#### Application (Storage)
- Kiểm tra:
  - Cookies
  - Local Storage
  - Session Storage
- Có thể tìm thấy:
  - Session Token
  - API Key
  - Tracking ID
  - Authentication Data

#### Security
- Xem thông tin Certificate:
  - Issuer
  - Validity
  - Subject Alternative Names (SANs)
- SANs có thể tiết lộ subdomain khác.

### Browser Extensions

| Extension | Chức năng |
|-----------|-----------|
| **FoxyProxy** | Chuyển đổi Proxy (Burp Suite, ZAP, SOCKS5). |
| **User-Agent Switcher** | Thay đổi User-Agent để giả lập trình duyệt/thiết bị khác. |
| **Wappalyzer** | Xác định Technology Stack (CMS, Web Server, Framework, CDN, Analytics...). |
| **BuiltWith** | Technology Fingerprinting tương tự Wappalyzer. |
| **WhatRuns** | Phiên bản nhẹ của Wappalyzer. |
| **Library Detector** | Phát hiện JavaScript Library và Framework. |

### Lưu ý
- Dù dùng Browser, hành vi bất thường vẫn có thể bị phát hiện:
  - Load trang quá nhanh.
  - Header bất thường.
  - Thay đổi User-Agent liên tục.
  - Hành vi khác thường.
- Mục tiêu là **mô phỏng người dùng hợp pháp** để giảm khả năng bị phát hiện.

## 3. Ping

### Khái niệm
- **Ping** dùng để kiểm tra xem host có **online** và **có thể kết nối** qua mạng hay không.
- Hoạt động dựa trên giao thức **ICMP**.

### Cách hoạt động
- Gửi **ICMP Echo Request (Type 8)**.
- Nếu host cho phép, sẽ phản hồi **ICMP Echo Reply (Type 0)**.

### Cú pháp

**Linux / macOS**
```bash
ping -c 5 <IP/Hostname>
```

**Windows**
```bash
ping -n 5 <IP/Hostname>
```

**Chỉ định IPv4 / IPv6**
```bash
ping -4 -c 5 <IP>
ping -6 -c 5 <IPv6>
```

### Phân tích kết quả

#### Ping thành công
- Host đang **online**.
- Có thể kết nối.
- `0% packet loss` → đường truyền tốt.
- RTT thấp → thường ở gần (cùng LAN hoặc ít hop).

#### TTL (Time To Live)
- Là **số hop tối đa** gói tin được phép đi qua.
- Mỗi Router sẽ giảm TTL đi **1**.

**TTL mặc định**
- Linux: **64**
- Windows: **128**

> TTL nhận được thấp hơn do đã đi qua nhiều router.
>
> Ví dụ:
> - TTL = **58** → Có thể là Linux cách khoảng **6 hop**, **không phải** Linux có TTL mặc định là 58.

### Ping thất bại

**Destination Host Unreachable**
- Host tắt.
- Không có route đến host.
- Máy chưa khởi động.

**100% Packet Loss**
- ICMP bị Firewall/WAF/CDN chặn.
- NAT chặn ICMP.
- Windows Firewall mặc định chặn Ping.
- Cloud (AWS, Azure, GCP) thường chặn ICMP.

### Quick Reference

| Kết quả | Ý nghĩa | Bước tiếp theo |
|---------|---------|----------------|
| Reply nhanh, ít hoặc không mất gói | Host online | Port Scan |
| Destination Host Unreachable | Host down hoặc không có route | Kiểm tra máy đích |
| 100% Packet Loss | ICMP bị chặn | TCP/UDP Discovery (Nmap) |
| Latency cao / Mất gói nhiều | Mạng nghẽn hoặc xa | Dùng Traceroute |

## 4. Traceroute

### Khái niệm
- **Traceroute** dùng để xác định đường đi của gói tin từ máy nguồn đến máy đích.
- Hiển thị:
  - Các **Router (Hop)** trung gian.
  - Số lượng **Hop**.
  - Độ trễ (Latency) của từng Hop.
- Hữu ích để:
  - Hiểu topology mạng.
  - Xác định nơi xảy ra độ trễ hoặc bị chặn.
  - Mapping hạ tầng mạng.

> **Lưu ý:** Đường đi không cố định do:
> - Dynamic Routing (BGP, OSPF)
> - Load Balancing
> - Anycast Routing

### Cú pháp

**Linux / macOS**
```bash
traceroute <IP>
```

**Windows**
```cmd
tracert <IP>
```

**IPv6**
```bash
traceroute -6 <IPv6>
traceroute6 <IPv6>
```

### Cách hoạt động
- Dựa trên trường **TTL (Time To Live)**.
- Gửi packet với TTL tăng dần:
  - TTL = 1 → Router đầu tiên trả về **ICMP Time Exceeded**.
  - TTL = 2 → Router thứ hai trả lời.
  - ...
  - Tiếp tục đến khi tới host đích.

### Kết quả đầu ra

- Mỗi dòng = **1 Hop**.
- Mặc định gửi **3 packet/Hop**.
- Hiển thị:
  - IP hoặc Hostname của Router.
  - RTT (Round Trip Time).

#### Dấu `*`
Có thể do:
- Router không phản hồi ICMP.
- Firewall chặn.
- Rate Limiting.
- Router được cấu hình ẩn.

#### Số Hop
- Hop cuối cùng = Host đích.
- Các Hop trước = Router trung gian.

### Traceroute trên Linux

Mặc định:
- Dùng **UDP**.

Có thể chuyển sang:

```bash
traceroute -T <IP>
```
- TCP Mode (vượt qua một số UDP Filter).

```bash
traceroute -I <IP>
```
- ICMP Mode.

### Lưu ý
- Số Hop có thể thay đổi giữa các lần chạy.
- Router xuất hiện cũng có thể khác.
- Nguyên nhân:
  - Dynamic Routing
  - Load Balancing
  - Failover
  - CDN (Cloudflare, Akamai)

### Công cụ liên quan

#### mtr (My Traceroute)
```bash
mtr <IP>
```
- Kết hợp **Ping + Traceroute**.
- Theo dõi:
  - Packet Loss
  - Latency
  - Đường đi theo thời gian thực.

### Traceroute dùng UDP hay ICMP?

- **Traceroute** hoạt động dựa trên trường **TTL**.
- Khi TTL về `0`, router trung gian sẽ trả về gói **ICMP Time Exceeded**.

#### Trên Linux
```bash
traceroute <IP>
```
- Mặc định gửi **UDP packet**.
- Router phản hồi bằng **ICMP Time Exceeded**.

```bash
traceroute -I <IP>
```
- Ép traceroute gửi **ICMP Echo Request**.
- Giống cơ chế của `ping`.

```bash
traceroute -T <IP>
```
- Ép traceroute gửi **TCP packet**.
- Hữu ích khi UDP/ICMP bị firewall chặn.

#### Trên Windows
```cmd
tracert <IP>
```
- Mặc định dùng **ICMP Echo Request**.

> Tóm tắt:  
> Linux `traceroute` mặc định = UDP  
> Linux `traceroute -I` = ICMP  
> Linux `traceroute -T` = TCP  
> Windows `tracert` mặc định = ICMP

## 5. Telnet

### Khái niệm
- **TELNET** là giao thức dùng để điều khiển hệ thống từ xa qua CLI.
- Port mặc định: **23**.
- Telnet gửi dữ liệu dạng **cleartext** → username/password có thể bị nghe lén.
- Thay thế an toàn hơn: **SSH**.

### Telnet trong Reconnaissance
- Có thể dùng Telnet để kết nối tới **bất kỳ TCP port** nào.
- Mục tiêu: đọc phản hồi ban đầu của service.
- Kỹ thuật này gọi là **Banner Grabbing**.

### Banner Grabbing là gì?
- Kết nối tới một service.
- Đọc thông tin service trả về.
- Banner có thể tiết lộ:
  - Tên phần mềm.
  - Phiên bản.
  - Loại web server/service.

### Ví dụ: Banner Grabbing HTTP port 80

```bash
telnet 10.49.160.55 80
```

Sau đó gửi HTTP request:

```http
GET / HTTP/1.1
host: telnet
```

Nhấn **Enter 2 lần** để gửi request.

Kết quả ví dụ:

```http
HTTP/1.1 200 OK
Server: nginx/1.6.2
Date: Tue, 17 Aug 2021 11:13:25 GMT
Content-Type: text/html
Content-Length: 867
Connection: keep-alive
```

Dòng quan trọng:

```http
Server: nginx/1.6.2
```

Ý nghĩa:
- Web server là **nginx**.
- Version là **1.6.2**.
- Có thể tra version này trên **CVE** hoặc **Exploit-DB** để kiểm tra lỗ hổng.

### Dùng với service khác
- **FTP port 21**: thường tự gửi banner ngay khi kết nối.
- **SMTP / POP3**: cần gửi command đúng giao thức.
- Nguyên tắc chung:

```md
Connect to port → Read banner → Gửi command phù hợp nếu cần
```

### Hạn chế
- Telnet không xử lý được kết nối mã hóa.
- Không phù hợp với:
  - HTTPS port 443
  - SMTPS port 465
  - Các service dùng TLS/SSL

### Công cụ thay thế

```bash
curl --head https://10.49.160.55
```

```bash
openssl s_client -connect 10.49.160.55:443
```

```bash
ncat --ssl <IP> <PORT>
```

### Ghi nhớ
- Telnet **không an toàn để đăng nhập từ xa**.
- Nhưng vẫn hữu ích để học:
  - TCP connection
  - Banner grabbing
  - Cleartext protocol interaction

## 6. Netcat (nc)

### Khái niệm
- **Netcat (nc)** là công cụ mạng đa năng hỗ trợ **TCP** và **UDP**.
- Có thể hoạt động như:
  - **Client** (kết nối đến server).
  - **Server** (lắng nghe kết nối).
- Ứng dụng:
  - Banner Grabbing
  - Port Probing
  - File Transfer đơn giản
  - Client-Server Communication

> So với Telnet, **Netcat linh hoạt hơn**, hỗ trợ IPv6 và SSL (qua `ncat`).

---

### Banner Grabbing

Kết nối đến web server:

```bash
nc 10.48.164.150 80
```

Gửi HTTP Request:

```http
GET / HTTP/1.1
host: netcat
```

> Có thể cần nhấn **Shift + Enter** sau dòng `GET / HTTP/1.1`, rồi nhấn **Enter 2 lần** để gửi request.

Ví dụ kết quả:

```http
HTTP/1.1 200 OK
Server: nginx/1.6.2
Content-Type: text/html
...
```

Dòng quan trọng:

```http
Server: nginx/1.6.2
```

→ Xác định:
- Web Server: **nginx**
- Version: **1.6.2**

Có thể tra trên:
- CVE
- Exploit-DB

---

### Banner Grabbing với service khác

FTP:

```bash
nc <IP> 21
```

→ Thường nhận banner ngay khi kết nối.

SMTP:

```bash
nc <IP> 25
```

→ Nhận banner của Mail Server.

Nguyên tắc:

```md
Connect → Read Banner → Gửi command đúng giao thức nếu cần
```

---

### Listen Mode

Server:

```bash
nc -vnlp 1234
```

Client:

```bash
nc <IP> 1234
```

Sau khi kết nối:
- Hai bên có thể gửi và nhận văn bản trực tiếp.

---

### Các tùy chọn thường dùng

| Option | Ý nghĩa |
|---------|---------|
| `-l` | Listen Mode |
| `-p` | Chỉ định Port |
| `-n` | Không phân giải DNS |
| `-v` | Verbose |
| `-vv` | Very Verbose |
| `-k` | Tiếp tục lắng nghe sau khi client ngắt |

> **Lưu ý:** `-p` phải đứng ngay trước số port.

---

### IPv6 & SSL

Listen IPv6:

```bash
nc -6 -lp 1234
```

SSL:

```bash
ncat --ssl <IP> <PORT>
```

---

### Ghi nhớ

- **Telnet**: chủ yếu dùng để học Banner Grabbing và giao thức Cleartext.
- **Netcat**: công cụ mạng đa năng, thường được ưu tiên hơn Telnet trong thực tế.
- Có thể dùng để:
  - Banner Grabbing
  - Kiểm tra kết nối TCP/UDP
  - Chat giữa 2 máy
  - Truyền file đơn giản
