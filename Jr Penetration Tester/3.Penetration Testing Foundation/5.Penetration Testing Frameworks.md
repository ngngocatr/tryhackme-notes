# Penetration Testing Framworks
## 1. Introduction
- Penetration Testing Frameworks là một phương pháp luận có cấu trúc, hướng dẫn các chuyên gia bảo mật trong mọi giai đoạn của một cuộc kiểm thử, từ lập kế hoạch ban đầu (initial planning), xác định phạm vi (scoping), khai thác (exploitation), lập báo cáo (reporting) cho đến xác minh việc khắc phục (remediation validation)
- Lợi ích:
    - Đảm bảo tính toàn diện (thoroughness): giúp không bỏ sót các khu vực quan trọng cần kiểm tra.
    - Đảm bảo tính nhất quán (consistency): giúp các kiểm thử viên khác nhau trong cùng một nhóm tạo ra kết quả tương đương nhau.
    - Hỗ trợ tuân thủ (compliance): giúp cuộc đánh giá phù hợp với các yêu cầu của quy định và tiêu chuẩn.
    - Cải thiện giao tiếp (communication): vì khách hàng, kiểm toán viên và các bên liên quan có thể hiểu và tin tưởng vào một quy trình dựa trên các tiêu chuẩn đã được công nhận.

- Hiện nay có nhiều framework kiểm thử xâm nhập, mỗi framework có triết lý, điểm mạnh và mục đích sử dụng riêng.

    - **OSSTMM**: Kiểm thử bảo mật dựa trên phương pháp khoa học và số liệu.
    - **OWASP WSTG**: Tiêu chuẩn phổ biến để kiểm thử ứng dụng web.
    - **NIST SP 800-115**: Hướng dẫn kiểm thử và đánh giá bảo mật của chính phủ Hoa Kỳ.
    - **PTES**: Quy trình pentest theo từng giai đoạn, sát với thực tế.
    - **ISSAF**: Framework lâu đời với mô hình đánh giá gồm 9 bước.

## 2. OSSTMM

### Overview

- Phát triển bởi **ISECOM**.
- Áp dụng **phương pháp khoa học** vào kiểm thử bảo mật.
- Đánh giá dựa trên **metrics** thay vì ý kiến chủ quan.
- Kết quả **định lượng (quantifiable)**, **kiểm chứng được (verifiable)** và **có thể lặp lại (repeatable)**.

### 5 Security Channels

| Channel | Mục tiêu |
|---------|----------|
| **HUMSEC** | Con người (Social Engineering) |
| **PHYSSEC** | Bảo mật vật lý |
| **SPECSEC** | Mạng không dây (Wi-Fi, Bluetooth, RFID...) |
| **COMSEC** | Viễn thông (VoIP, điện thoại...) |
| **DATASEC** | Mạng dữ liệu (Network, Firewall, Protocols) |

> Mục tiêu: Không bỏ sót bất kỳ khía cạnh bảo mật nào.

### Risk Assessment Value (RAV)

- Đo mức cân bằng giữa:
  - **Attack Surface (Exposure)**
  - **Security Controls**
- **RAV > 0:** Vẫn còn rủi ro.
- **RAV ≈ 0:** Mức bảo vệ phù hợp với mức phơi bày.

### Testing Phases

#### 1. Induction
- Enumeration
- Verification
- **Output:** Danh sách tài sản đã xác minh.

#### 2. Interaction
- Fingerprint services.
- Đo mức độ phơi bày (Exposure).
- Tính **Attack Surface**.

#### 3. Inquiry
- Khai thác lỗ hổng.
- Kiểm tra khả năng leo thang đặc quyền.
- Xác minh phạm vi ảnh hưởng.

#### 4. Intervention
- Quarantine (Khắc phục/Cô lập).
- Audit (Rà soát hệ thống).
- Enticement (Canary Token/Honeypot để kiểm tra khả năng phát hiện).

### Reporting

- Chuẩn báo cáo: **STAR (Security Test Audit Report)**.

### Pros

- Kết quả khách quan, định lượng.
- Có thể kiểm toán.
- Có thể so sánh giữa các lần đánh giá.
- Tính nhất quán cao.

### Cons

- Khó học.
- Triển khai mất nhiều thời gian.
- Ít chuyên gia hơn OWASP và NIST.

### Best Use Case

Phù hợp với tổ chức cần:
- Đánh giá bảo mật có thể **đo lường**.
- Kết quả **lặp lại** và **kiểm toán** được.
- Đánh giá mang tính khoa học.

## 3. OSWASP WSTG

### Overview

- **OWASP Web Security Testing Guide (WSTG)** là framework kiểm thử bảo mật ứng dụng web.
- Được phát triển bởi cộng đồng OWASP.
- Gồm hơn **90 test cases**, chia thành **12 nhóm**.
- Mỗi test case có mã định danh, mục tiêu và hướng dẫn kiểm thử cụ thể.
- Áp dụng **risk-based approach**, ưu tiên lỗ hổng dựa trên:
  - Khả năng khai thác (exploitability)
  - Mức độ ảnh hưởng (impact)

### 12 Testing Categories

- Information Gathering
- Configuration & Deployment Management
- Identity Management
- Authentication
- Authorization
- Session Management
- Input Validation
- Error Handling
- Cryptography
- Business Logic
- Client-side Testing
- API Testing

### Security Across the SDLC

#### Phase 1: Before Development
- Xác định yêu cầu bảo mật.
- Xác định tiêu chuẩn cần tuân thủ, ví dụ PCI DSS.

#### Phase 2: Definition & Design
- Review kiến trúc.
- Threat Modeling.
- Thiết kế security controls từ sớm.

#### Phase 3: Development
- Code Review.
- Code Walkthrough.
- Đối chiếu mã nguồn với WSTG test cases.

#### Phase 4: Deployment
- Penetration Test trên môi trường staging/production.
- Kiểm tra:
  - Default credentials
  - TLS configuration
  - Debug endpoints
  - Security controls

#### Phase 5: Maintenance & Operations
- Kiểm tra bảo mật định kỳ.
- Re-test sau khi cập nhật tính năng.
- Đảm bảo thay đổi mới không tạo thêm lỗ hổng.

### Advantages

- Bao phủ gần như toàn bộ bề mặt tấn công web.
- Có checklist và test case rõ ràng.
- Giảm nguy cơ kiểm tra trùng hoặc bỏ sót.
- Phù hợp cho pentest web, code review và SDLC.
- Giúp ưu tiên các lỗ hổng quan trọng.

### Best Use Case

Phù hợp với:

- Web application penetration testing
- API security testing
- Secure SDLC
- Code review
- Security assessment trước và sau triển khai
